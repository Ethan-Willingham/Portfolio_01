The Box Shadow Sponge

23rd of April, 2026

Last year David Gerrells published How Not to Use Box Shadows, a piece that escalated from playful CSS experiments to a full ray tracer rendered entirely through the box-shadow property of a single <div>. It's one of the most unhinged and delightful things I've read about CSS.

The whole article is worth your time, but one demo stopped me cold: the sponge. A cluster of colored circles, each drifting gently to a home position, that you can reach in and tear apart with your cursor. Release, and they drift back. It looks organic — like pulling bits off a sea sponge. I wanted to understand exactly how it works, so I rebuilt it from scratch.

There is no <canvas> in that demo. No WebGL. No SVG. Every single one of those circles is a box-shadow value on a 1×1 pixel <div>. The browser's CSS rendering pipeline is doing all the drawing.

How a box-shadow becomes a particle

The CSS box-shadow property lets you stack an arbitrary number of shadows on a single element, separated by commas. Each shadow takes an x offset, y offset, optional blur radius, optional spread radius, and a color. When you set blur to zero and give it a spread, you get a hard-edged circle (if the element has border-radius: 50%) positioned wherever you want.

The trick is to make the element tiny — one pixel — and let the spread radius control the visual size. A shadow with 200px 150px 0 8px #A8B545 draws a green circle of radius 8 at coordinates (200, 150). Stack 120 of those, each with its own position and color, and you have a particle system. 

/* The entire "renderer" — build a box-shadow string each frame */
function render() {
  balls.sort(function (a, b) { return b.z - a.z; });

  var parts = [];
  for (var i = 0; i < balls.length; i++) {
    var b = balls[i];
    var zScale = 1 + b.z / 40;
    var radius = b.size * zScale;
    var spread = (radius - RENDER_SIZE) / 2;
    parts.push(
      b.x + 'px ' +
      b.y + 'px 0 ' +
      spread + 'px ' +
      b.color
    );
  }
  renderDot.style.boxShadow = parts.join(',');
}

That is the entire render function. Every frame, sort by depth, compute each particle's spread from its size and a fake z-scale, join the array into a comma-separated string, set it on the element. Done. The browser does the rest.

The sponge behavior

Each ball has a home position — a random coordinate assigned at initialization. Every frame, a spring force pulls the ball back toward home. This is a basic spring equation: the farther the ball is from home, the stronger the pull. 

/* Spring toward home */
var springK = 1.8;
var dx = b.homeX - b.x;
var dy = b.homeY - b.y;
b.vx += dx * springK * dt;
b.vy += dy * springK * dt;

When the pointer is active, balls within a 200px radius get pulled toward it instead. The pull is strongest at the center and falls off linearly. Because the pull force is much larger than the spring force, nearby particles rip away from their home positions — the sponge tears. 

/* Pointer pull */
if (pointerDown) {
  var px = pointerX - b.x;
  var py = pointerY - b.y;
  var dist = Math.sqrt(px * px + py * py);
  if (dist < 200 && dist > 1) {
    var pullStrength = (1 - dist / 200) * 900;
    b.vx += (px / dist) * pullStrength * dt;
    b.vy += (py / dist) * pullStrength * dt;
  }
}

When you release, the pull disappears and the springs take over. Velocity damping (b.vx *= 0.94) bleeds off energy so the balls don't oscillate forever. The result is that satisfying drift back into formation.

The fake 3D

Each ball also has a z value. This doesn't affect position on screen — it's purely cosmetic. The z value scales the ball's visual size through a simple ratio: 1 + z / 40. Balls with a higher z look "closer" (larger), lower z looks "further" (smaller). Sorting by z before rendering means distant balls are drawn first, naturally occluded by nearer ones.

It's not real perspective projection. There's no vanishing point, no foreshortening. But it's enough to give the cluster a sense of depth, especially when the balls are in motion and the z-perturbation adds a bit of parallax-like wobble.

Performance notes

Gerrells noticed something interesting about box-shadow performance on Apple Silicon: without blur or transparency, the browser can push thousands of box-shadows at interactive frame rates. Blur is expensive because it requires a convolution pass. Transparency is surprisingly expensive too, likely because the compositor has to blend layers rather than just overwriting pixels.

The demo above avoids both — every color is fully opaque, blur is zero. On an M-series Mac you'll see a solid 60fps with 120 particles. On older hardware or low-end phones you might drop to 30-40, which still feels fine for this kind of organic motion.

The size of the source element matters too. A larger element means the browser has to rasterize a larger initial box before applying shadows. By keeping the element at 1×1 pixel, we minimize that base cost and let spread do all the work.

Things that surprised me

String concatenation is the bottleneck. Profiling the demo, the most expensive operation per frame is not the physics update or the sort — it's building the box-shadow string. Joining 120 entries with coordinate values creates a string on the order of 5-6KB every single frame. The browser then has to parse that string back into its internal representation. For Gerrells' image demos (12,000+ particles), the shadow string itself becomes enormous.

The browser doesn't cache shadow layouts. Every time you set element.style.boxShadow, the browser treats it as a completely new value and re-parses, re-layouts, and re-paints. There's no diff. This is partly why Gerrells found that transition: all was actually slower than raw requestAnimationFrame updates — the transition machinery adds overhead on top of something that's already being fully recalculated.

It works because GPUs love rectangles. Box shadows are fast specifically because they map to the same rounded-rectangle primitives that GPUs are already optimized to draw for UI compositing. A box-shadow is essentially "draw another rounded rect at this offset with this color," which is exactly the kind of operation that hardware acceleration handles well. The moment you add blur, the GPU has to do texture sampling, and the cost profile changes dramatically.

The full code

Below is the complete, self-contained simulation script. No dependencies, no build step. It expects a container element with id="sponge-demo" containing a child with class render-dot — a 1px element with border-radius: 50%. The CSS for the demo container handles sizing and cursor styles. 

(function () {
  'use strict';

  var container = document.getElementById('sponge-demo');
  if (!container) return;

  var renderDot = container.querySelector('.render-dot');
  var PALETTE = [
    '#A8B545', '#7A9B3A', '#C2CC68', '#556B2F',
    '#8B6E4E', '#6B4226', '#A0522D', '#D2B48C',
    '#C4B38A', '#EDE6DA', '#8F8978', '#5E6B42'
  ];

  var W, H, balls = [], BALL_COUNT = 120;
  var pointerDown = false, pointerX = 0, pointerY = 0;
  var prevTime = 0;

  function resize() {
    var rect = container.getBoundingClientRect();
    W = rect.width; H = rect.height;
  }

  function createBalls() {
    balls = [];
    for (var i = 0; i < BALL_COUNT; i++) {
      var hx = Math.random() * W, hy = Math.random() * H;
      balls.push({
        x: hx, y: hy, z: Math.random() * 60,
        homeX: hx, homeY: hy,
        vx: 0, vy: 0, vz: 0,
        size: 4 + Math.random() * 14,
        color: PALETTE[Math.floor(Math.random() * PALETTE.length)]
      });
    }
  }

  function update(dt) {
    dt = Math.min(dt, 0.05);
    for (var i = 0; i < balls.length; i++) {
      var b = balls[i];
      b.vx += (b.homeX - b.x) * 1.8 * dt;
      b.vy += (b.homeY - b.y) * 1.8 * dt;
      if (pointerDown) {
        var px = pointerX - b.x, py = pointerY - b.y;
        var dist = Math.sqrt(px * px + py * py);
        if (dist < 200 && dist > 1) {
          var f = (1 - dist / 200) * 900;
          b.vx += (px / dist) * f * dt;
          b.vy += (py / dist) * f * dt;
        }
      }
      b.vx *= 0.94; b.vy *= 0.94;
      b.x += b.vx * dt; b.y += b.vy * dt;
    }
  }

  function render() {
    balls.sort(function (a, b) { return b.z - a.z; });
    var s = [];
    for (var i = 0; i < balls.length; i++) {
      var b = balls[i];
      var spread = (b.size * (1 + b.z / 40) - 1) / 2;
      s.push(b.x+'px '+b.y+'px 0 '+spread+'px '+b.color);
    }
    renderDot.style.boxShadow = s.join(',');
  }

  function tick(t) {
    update((t - prevTime) / 1000);
    prevTime = t;
    render();
    requestAnimationFrame(tick);
  }

  // pointer events...
  container.addEventListener('mousedown', function(e) {
    pointerDown = true;
    var r = container.getBoundingClientRect();
    pointerX = e.clientX - r.left;
    pointerY = e.clientY - r.top;
  });
  // ... (touch events similar)

  resize(); createBalls();
  prevTime = performance.now();
  requestAnimationFrame(tick);
})();

The palette in the demo uses earthy greens and browns — close to the forest-green background of this site. The original article uses a brighter, more saturated set. Swap the PALETTE array for any colors you like; the physics don't care.

Why this matters (a little)

Gerrells' project is delightful partly because it's impractical. Nobody should ship a production particle system powered by box-shadow strings. Canvas and WebGL exist for a reason. But the exercise reveals something about how browsers actually work under the hood: the rendering pipeline is more capable and more flexible than most developers ever test. Box-shadow wasn't designed for this. It works anyway. That gap between intended use and actual capability is where all the interesting creative work happens.

The article also ends up being a surprisingly good introduction to real-time rendering concepts — game loops, delta time, z-sorting, spring physics, progressive ray tracing — all presented through the lens of an absurd constraint. Sometimes the best way to learn the serious version of something is to see the joke version first.

The original article and all demos are at dgerrells.com. The source code for the more advanced demos (including the ray tracer) is on CodeSandbox.
