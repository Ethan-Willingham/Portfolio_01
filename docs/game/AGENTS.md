## What this project is

A browser game (`Sluice`). Vanilla HTML/JS/CSS, served as-is. To test, open `sluice.html` in a browser (or use the `preview_*` tools).

The game is a 2D mining game written entirely with LLMs across multiple accounts (3 Claude + 1 Codex/GPT). The owner switches accounts frequently — *consistency between sessions matters more than micro-optimization.* When in doubt, match what's already there.

**Path note:** HTML files reference scripts as `js/filename.js` and fonts as `assets/fonts/filename.woff2`. `style.css` uses `assets/fonts/` for `@font-face`.

**Fonts — the three-font rule (load-bearing):** the entire site uses exactly THREE typefaces, all self-hosted in `assets/fonts/` and declared via `@font-face` in `style.css`. No external font CDN (no Google Fonts), no fourth face.
- **Century Supra** (serif) — headings + editorial voice. `var(--font-heading)`.
- **Segoe UI** (humanist sans) — body + UI. `var(--font-body)` (local()-first, with self-hosted `.woff` fallback).
- **Commit Mono** (monospace) — everything monospaced: code, data readouts, gauges, dev panels, and the game HUD. Reached via `var(--font-mono)`, `var(--d-mono)`, and the game's `UI_FONT` constant in `sluice.js` (canvas, so `sluice.html` preloads it). All of these resolve to `'Commit Mono', ui-monospace, Consolas, monospace`.

When you add or touch any text, keep it on one of these three. Monospace is always Commit Mono; do not reintroduce SF Mono / JetBrains / system-mono stacks or pull a font from a CDN.

**Physics is GPU-first (WebGPU MLS-MPM).** The water/oil simulation runs as **WebGPU compute** in `js/liquid-wgpu.js` (a staged MLS-MPM port, LIVE since v14.0 when the device supports WebGPU). The `liquid*` APIC code inside `sluice.js` is its **CPU fallback + bit-faithful self-test reference** — don't read it as "physics is CPU-only" (the old "the fix is WASM, not GPU" banner predated the v14.0 GPU port and was corrected in v24.187; the GPU WAS the fix, never chase WASM). **Prefer WebGPU/GPU for physics where possible.** `TUNING.md` §2.9 lists the WebGPU water levers and **§2.10 the stability / anti-clump fix**; any lever marked `edit²` is duplicated in `liquid-wgpu.js` — change both copies or the GPU sim drifts from the CPU reference.

**Do NOT save screenshot PNGs to this directory.** Past sessions left behind 30+ `*-verify.png` and `*-smoke-*.png` debug screenshots that were deleted. Use `preview_screenshot` to view visuals during a session, but do not write them to disk as project files.

## Split layout: `js/sluice/` fragments

The game is split across **fragment files in `js/sluice/`** so a session can open the one piece it needs (~a few hundred to ~2.5k lines) instead of re-orienting in a 38k-line file every time. **`js/sluice.js` is a generated bundle**, not the source.

**How it works (dumb text-concat, fully reversible):**
- The fragments are named `NNN-*.js` (zero-padded numeric prefix). Each is a **raw text slice of the one IIFE**: `000-head.js` opens it, `999-tail.js` closes it (`})();`), every fragment in between is the body in order. A middle fragment is **not** valid JS on its own; only the assembled bundle is.
- `./build-sluice.sh` concatenates `js/sluice/[0-9][0-9][0-9]-*.js` in numeric order into `js/sluice.js`. No transform, no minify, so the bundle is byte-for-byte the sum of its parts, stays plain readable JS, and `sluice.html` loads it unchanged.
- **Both are committed.** Fragments are the editable source; `js/sluice.js` is committed too because Amplify serves it directly.

**The reversibility invariants (do NOT break these, or the split stops being a lossless concat):**
- **No `import` / `export`.** Fragments share one scope; they don't module-import each other.
- **No per-file IIFE / no `'use strict'` per fragment.** There is exactly one wrapper, in `000-head.js` + `999-tail.js`.
- **No namespace object** (`var S = {}; S.foo = …`). Functions and `var`s are declared at IIFE top-level exactly as before; hoisting means a fragment can call something defined in a later fragment.
- Keep everything that was already an invariant: `var` not `let`/`const`, single IIFE, no deps.

**Workflow:**
1. Edit the relevant fragment in `js/sluice/` (find it in the map below, or grep the bundle to see which section, then open that fragment).
2. Run `./build-sluice.sh` to regenerate `js/sluice.js`.
3. Bump `GAME_VERSION` (it lives in `000-head.js`), then commit BOTH the fragment(s) and the rebuilt `js/sluice.js`, and push (see "Versioning & save rules").
- **Never hand-edit `js/sluice.js`**: the next build overwrites it. Edit the fragment.
- **To collapse back to one hand-edited file** (undo the split entirely): run `./build-sluice.sh` once so the bundle is current, then `rm -rf js/sluice build-sluice.sh`. `js/sluice.js` is untouched by that removal and becomes your single source file again. Nothing else changes.

## Code style — match the existing file

These aren't preferences, they're invariants. Don't "modernize" them.

- **`var` everywhere.** Not `let` or `const`. The whole file uses `var` with hoisted scope. Mixing in `let`/`const` mid-file is jarring.
- **Single IIFE wrapper.** All game code lives inside one `(function () { 'use strict'; … })();`, opened in `000-head.js`, closed in `999-tail.js`, with the fragments as raw slices of the body. Don't introduce ES modules or a per-fragment wrapper (see "Split layout").
- **No transpile-only syntax.** No JSX, no TypeScript annotations, no top-level await.
- **The game is ONE IIFE split into raw text fragments, not modules.** Adding a new `js/sluice/NNN-*.js` fragment (or splitting one that has grown) is fine and encouraged when a section gets unwieldy; that's the point of the layout. What stays forbidden is turning fragments into real modules: no `import`/`export`, no per-fragment IIFE, no namespace object (see "Split layout" for why). Don't add new top-level files outside `js/sluice/` for game code.
- **Function-scoped helpers** are fine; prefer them to splitting files.
- **Keep the existing comment style** — `/* ---- Section ---- */` for major sections, `// ----- Subsection -----` for inner blocks, `// ====== BIG BANNER ======` for the few top-level systems (rover, explosives, UI space). Sentence-case English. The existing comments are good — match their voice.

## Invariants & gotchas

Things that look weird but are intentional. Don't "fix" them without checking.

### Ore HP / drill levels
- `hp` on an ore is the number of drill hits it takes — one hit removes 1 HP, regardless of drill level. Each hit takes `DRILL_TIME / DRILL_SPEED[drillLevel]` (base `DRILL_TIME` = 0.30s at level 1), so a higher drill tier drills each hit FASTER (see the `DRILL_SPEED` table in the constants block) while the hit COUNT stays = hp. (Pre-v23.17 the drill subtracted `drillLevel` HP per hit, so `ceil(hp/level)` left most tiers doing nothing on common ores — that integer model is gone; `drillHitTime()` is the helper.)
- `reqDrill` is the *minimum drill upgrade level* to mine it. Below that level, the drill **bounces** (visible animation), it doesn't just slow down.
- `reqHeat: true` means it needs the heated-drill upgrade. Same bounce behavior.
- HP `999999` means unbreakable by drill — only explosives clear it (e.g. `barrier`, `bedrock`).

### Barrier band
A reinforced row at the permafrost/fossil boundary. Drills bounce off; only bombs clear it. Spawned at fixed depths in `generateWorld()`, not by `pickOre`.

### Bedrock
- `BEDROCK_ROW` (last row of Earth world) is unbreakable and caps the bottom of the world.
- There is no win condition: the game is an open-ended mining sandbox (the moon, which was the old win, was removed in v23.18). `gameWon` survives as an inert, never-true flag so its internal guards still work; don't wire a win to it without checking those guards.

### Persisted state / SAVE SYSTEM (v24.93)
`localStorage` now holds:
- `sluice_devmode` — dev mode flag (backtick toggles).
- `sluice.zoom` — zoom mode.
- `sluice.volume` — pause-screen volume slider.
- `sluice.save.a` / `sluice.save.b` — **the game SAVES PROGRESS now** (persistent-profile model, `047-save.js`). Two rotating slots, newest valid wins. Worldgen is unseeded, so the grid is the source of truth — never try to re-derive it from generation.

In-run R double-press = same-economics bailout to town (`bailoutToTown`); dev mode R keeps the old full `init()`. ### Dev mode
Press <kbd>`</kbd> (backtick) in-game. Money clamps to 999,999, all shop purchases are free. Useful for quickly verifying late-game features without grinding. Mention this whenever you tell the user how to test deep-game changes.

**Perf-panel caveat (v13.13):** dev mode shows the perf overlay, but the GPU probes (`gpuProbe`, which call `gl.finish()`) are OFF by default. `gl.finish()` is a hard CPU↔GPU sync — leaving it on serialises the pipeline and the panel under-reports FPS by a wide margin (it cost ~15fps on the surface). Press <kbd>G</kbd> in dev mode to enable the smoke/sky/liquid GPU breakdown when you actually need it; expect the FPS reading to drop while it's on. **The honest framerate is the one with the GPU probe off.**

**PERF ISO (v13.14):** press <kbd>H</kbd> in dev mode to cycle the PERF ISO A/B tool — each press disables the next subsystem (smoke fluid → rocket plume → night sky → mountains → console gauges → terrain chunks) and toasts the mode, so a GPU hog can be isolated empirically by watching the FPS at each step. The `PERF_DISABLE_*` flags all default false, so the build is unaffected unless `H` is pressed.

### Flight: ONE model everywhere, no modes, no camera juice
The v25.49 rebuild deleted the three sky flight modes (legacy "Today" 0, full rotation 1, VTOL hover 2), the separate underground air envelope, the 3-block `flightDeepUnder` handoff band, the pause-screen toggle, the dev F hotkey, and the mobile split rotate/thrust pad. What remains is **one integrator** (080) driven by **`flyTune`** (020, the `fly` gm group): upright rig, axis-aligned thrust, identical constants above ground and underground. The vertical model is the proven jet stack (tap kick, spool, headroom envelope toward `climbTerm`, hover assist, gravity relief, apex easing, hover settle) plus one new asymmetry: **`catch`**, a thrust multiplier that applies only while FALLING, so a long fall arrests in ~0.9 s (a real inertia fight, not the old ~3 s underground crawl) while the launch stays gentle (~166 px/s sustained climb). Horizontal is direct authority with a reversal bite (`revBoost`), a cruise cap that never confiscates earned overspeed (`overBleed` bleeds it), and `fric` on release. **`FLY_PRESETS`** (Default / Floaty / Freight / Miner) sit in the FLY FEEL strip pinned atop the L panel; `Miner` restores the old tight underground horizontal if the unified 290 px/s cruise feels hot in tunnels. On touch the plain dig d-pad drives flight in every regime (touch hit-tests stay **`isMobile`-gated; don't un-gate them or desktop clicks move the rig**). The 5-tier **booster** (tier 1 = the 1.0x baseline; `BOOST_THRUST_MULT`, 040) scales `climbForce` EVERYWHERE, so every tier is a felt climb + catch upgrade (the old v24.59 "sky is upgrade-independent" rule retired with the modes); the per-tier exhaust is unchanged (`BOOST_FLAME`, 200). **Owner rule: the flight feel is PURE physics, zero camera motion** (a look-ahead was tried + removed). All levers + the feel math are in **TUNING.md §6.2-6.3**; read that before touching flight.

### Underground wall pattern — occlusion architecture (v13.11/v13.12)
The biome wall is a **parallax background layer occluded by the terrain**, not a masked overlay (BACKGROUND_STYLE §10). It replaced a per-chunk clip-and-fill pass that cost ~200 complex `Path2D` fills per frame.
- **The wall** is painted in the undergroundBg pass — one parallaxed `fillRect` per visible biome band, full screen width, BEHIND the terrain chunks. `getBiomeWallFill(layerName)` returns a cached repeating `CanvasPattern`; the X+Y parallax rides on the pattern's own `setTransform` matrix (`BIOME_WALL_PARALLAX_X/Y`).
- **The terrain occludes it.** `drawSmoothVoids` (inside `renderTerrainChunk`, cached) makes each chunk opaque everywhere except the smooth cave contour: step 1 `destination-over`-floods a rock backing behind the tiles so empty cells become opaque; step 2 `destination-out`-erases the `buildVoidContourPath` marching-squares contour. The visible cave is the terrain's transparent negative space — no per-chunk masking.
- **Load-bearing invariant:** step 1 (the backing flood) is mandatory. Empty cells render to nothing, so without it the `destination-out` erase is a no-op where cells are already empty and the cave reverts to the blocky tile grid (right angles — the v13.11 bug). Do not remove the backing flood.
- Wall pattern source canvases (64×64 world px) are built once per biome by `buildSubtleWallPattern(seed, wallColor, brightAccent)` — solid wall colour + ~6 darker `wallMortar` specks + ~3 brighter biome-bg accent specks. **No grid, no organic loose-material features** — the wall must read as distinct shape language from the foreground material or it fails the §2 "lower detail frequency" test.
- The terrain chunk cache uses key format `chunkR + ':' + chunkC` via the `terrainChunkKey` helper — use the helper, never build the key by hand.
- Magma + mantle skip the wall — `getBiomeWallFill` returns null and the undergroundBg pass draws their heat gradient + embers instead (revealed through the erased voids).

### Mountain rendering — strip cache + per-peak invariants
**Strip cache (v13.17/v13.19):** the 3 mountain layers are static scenery that only parallax-scrolls, but path-drawing them live cost ~120 canvas path ops/frame (canvas path tessellation is slow). So each layer is rendered once into an offscreen strip (`buildMtnStrip` → `drawMtnPasses`) and blitted with one `drawImage` (`drawMountainLayer`). The strip rebuilds only on scroll/zoom out of range, or when `timeOfDay` crosses a `MTN_TIME_BUCKETS` bucket (the aerial tint + moon-rim track the day/night cycle) — at most one layer per frame. The blinking outpost lights animate, so `drawMtnLights` draws them live after the blit. **v13.15 lesson:** merging the per-peak paths into one `Path2D` fill *tripled* the cost (a complex self-overlapping fill) — the fix is to stop path-drawing static geometry every frame, not to batch the paths.

**Per-peak invariant (in `drawMtnPasses`):** each peak is **its own closed polygon** and **its own polyline stroke**, NOT a single continuous path through all peaks. Load-bearing — a single polygon self-intersects at overlapping peaks (per-peak asymmetric tilt) and the nonzero fill rule leaves bowtie holes; a single rim polyline draws connector lines through valleys. See BACKGROUND_STYLE §11 for the full ruleset (asymmetric tilt, sub-peak height variance, moon-side rim, no internal shadow polygons).

### Camera framing — depth-aware vertical
`updateCamera` computes `camFrac` from `player.y - surfaceY`:
- At/above surface: `SURFACE_FRAC = 0.55` (player slightly above centre, some sky visible above)
- 32 tiles below surface or deeper: `DEEP_FRAC = 0.50` (centered)
- Between: ease-in-out cubic
Pure function of player.y so flying back up reverses smoothly. Currently 0.55/0.50; was 0.72/0.50 in v10.46 before user requested less sky on screen. See BACKGROUND_STYLE §9.

Load-bearing invariants:
- **Lazy activation is the perf trick.** A buried jello tile is a free static tile with zero cost. It only becomes a live blob when `markTerrainCleared` fires on a cardinally adjacent cell (`jelloCheckActivation`). Do not make jello simulate while buried.
- **Clusters must stay 4-connected.** `traceJelloBoundary` relies on every boundary vertex having degree exactly 2 (true for any 4-connected cell set; a diagonal pinch is impossible). `generateJelloPatches` grows clusters by a 4-connected random walk; keep it that way or the tracer can mis-chain.
- **A settled blob is a free sleeping body, never re-tile it.** Converting a stopped blob back into grid tiles causes alignment hell. It stays a blob.
- **Jello is unminable and bomb-proof.** The drill bounces (`drillBlockReason`); a bomb blast activates + shoves it (`jelloBombShove`) rather than destroying it.
- **Drawn live on the entity layer**, never in the cached terrain chunks (jello is in the chunk-base exclusion lists alongside `barrier`/`bedrock`).
- The feel (underdamped wobble, slight over-inflation, squeeze-through-gaps) lives in the `JELLO_*` tunables block; tune there, do not re-architect.
- **Three switchable solvers (v21.30+; default `xpbd` since v21.37).** `JELLO_SOLVER` ('pbd' | 'xpbd' | 'fem'), cycled live with the `M` dev hotkey or `jello.JELLO_SOLVER_ID` lever (0/1/2). 'pbd' is the **jello-v1** baseline. 'xpbd' uses compliant distance + volume constraints with Lagrange multipliers (frame-rate-independent stiffness). 'fem' is XPBD-FEM: per-triangle Stable Neo-Hookean on the lattice mesh. Only the inner constraint solve swaps; all other systems (build, activation, sleep/freeze, player coupling, rendering) are shared. Roll back to the baseline by switching the solver to 'pbd' (the baseline code path ships in every build); the old `jello-v1` git tag did not survive the public-to-private repo split (this repo's history starts at v24.13; the tagged commit survives only in the Portfolio_01 archive, `portfolio-01-archive` + `~/portfolio-01-full-history.bundle`).
- The contact model is FleX-style per-particle (global count-sort spatial hash, inelastic positional separation + Coulomb friction); see the `JELLO SOFT BODIES` banner SOLVERS/CONTACT section in the code for the full algorithm. Validate any solver or contact change with a JXA numerical pile harness, because the preview cannot run the RAF loop.
- **Bulletproofing pass (v25.15) — do not undo these:**
  - **One name, one `var`.** The single-IIFE build means a re-declared `var JELLO_*` silently overwrites the tuned value at load (`JELLO_PUSH` shipped 0.7 instead of its documented-mandatory 1.05 this way, which is why walk-speed bulldozing stalled). Before adding any `JELLO_` var, grep the fragment for the name; a duplicate-declaration scan is part of the jello stress harness.
  - **Non-finite state can never reach the canvas or the rig.** `tileAt` survives NaN rows (returns open sky, never `world[NaN][c]` — that TypeError killed the whole game loop); `jelloWorldSolidAt` rejects non-finite coords; the contact-pair gate uses `!(d2 < rr*rr)` so NaN pairs skip; `jelloSanitizeBody` sweeps every active body once per frame (corrupt points heal to the finite centroid; an unrecoverable body despawns with a splat + console.warn); `jelloDrawBody` bails on a non-finite bbox (canvas gradients THROW on NaN); the ground probe zeroes non-finite ride velocities.
  - **Live bodies persist across save/load** (`jelloSaveBodies`/`jelloRestoreBodies`, the additive `jello` envelope field in 047): original cluster cells + type + centroid, rebuilt at rest pose, settle then sleep. Restore-side validation must accept everything save can write (dev-arena cells can sit a few rows above the world top) while bouncing corrupt payloads.
  - **World rebuilds reset the bodies** (`generateWorld` calls `resetJello()`; the live path never calls `generateJelloPatches`, which used to carry it) and the dev test pen is gated on `devMode && ENABLE_JELLO` (flag off = truly inert, no ghost bodies or undrillable pen tiles).
  - **Confined piles sit still; jets respect walls; corner pinches are sealed (v25.20).** The jet cone is occluded at the first SOLID tile along the exhaust (`jelloJetMaxS`, a solid-only ray — gel must never shield gel, walls always do). A point may never hop cells diagonally when BOTH orthogonal intermediates are solid (points used to thread the zero-width corner between two diagonal tiles). An overfilled pocket trips CROWD CALM (`JELLO_CROWD_CALM`: sustained heavy body-body contact drains velocity per frame) and its merged pairs PARK via a progress gate (no separation growth in ~40 frames), re-armed EVENT-driven when their geometry changes — timed retries kept some pair always mid-attempt. A long-deadlocked CONCENTRIC pair (2.5s total merged, centroids under a third of the separated distance) PHASES: the pair's mutual contact suspends while the ordinary rate shifts slide the interleaved lattices apart, then contact resumes. Do NOT replace phasing with a teleport pop (tried: destination-only probes wormed bodies through sealed walls, and even path-probed pops yanked press-loaded pairs — three harness scenarios regressed). ### Surface water resting calm (v24.125 baseline, LOCKED by owner)
A pond at rest must be GLASSY, including with the rig floating in it. Owner-approved acceptance floor: floating rig mid-pond = >90% of particles asleep, mean churn under ~2 px/s, max under ~10 px/s, zero wake bursts once the entry splash decays; shore pond = 100% asleep, max ~3 px/s. The resting-pond "firecrackers" (sections jolting in sync ~1/s) took many sessions to root-cause; they were THREE stacked player-coupling injectors, and these invariants keep them dead (all lockstep CPU + GPU):
- **The miner-silhouette eject is SPEED-GATED** (zero below 8 px/s rig speed, full at 60; `liquidApplyPlayerGridWake` in 070, WGSL `gridWake`, rig vx/vy ride GameParams counts.zw). It exists to clear water the rig PLOWS through; unconditional it is a constant 720 px/s² body force that drove a permanent ~143 px/s jet under a parked rig. Never make a player-coupled fluid force unconditional.
- **A static rig PINS in-silhouette grid cells to exact zero** (rigid boundary). Those cells hold splat mass but contain no particles and sit outside the terrain boundary handling; unpinned they free-fall under gravity forever and G2P feeds it back as a vertical jet (the old always-on eject had masked this). Any solid overlapping the fluid grid needs a real boundary condition.
- **The wake threshold must sit ABOVE the pond's measured ambient ripple peaks** (`LIQUID_WAKE_CELL_VSQ` = 0.007 ≈ 53 px/s; ambient peaks measure 40-90 px/s when energized). Below that, the water re-wakes its own sleepers forever and never converges. Real disturbances (swim eject, digs, bombs, fills) are hundreds of px/s or explicit wake ops.

**Regression harness — run before/after ANY change to water, the player-liquid coupling, sleep/wake constants, or the gridUpdate kernel:** serve the checkout (`python3 -m http.server 8799`), then headless Chrome (`--headless=new --enable-unsafe-webgpu --use-angle=metal --enable-logging=stderr`, fresh `--user-data-dir` per run) against `sluice.html?nosave=1&pondtest=1&wdbg=DBG_DRAW:1,DBG_READBACK_EVERY:6` (rig on shore) and the same with `pondtest=2` (rig FLOATING mid-pond, the pose that exposed all three pumps). The `POND:` console probe prints sleep counts / mean / max / wake-burst log every ~5 s; hold the acceptance numbers above. `?wdbg=NAME:V,...` sets any water gm lever at boot. In-game: dev hotkey `X` cycles the water debug overlay + per-mechanism kill switches; the on-screen meter shows the same stats live.

### Surface water stability — the giant-particle / runaway fix (v24.182-186, SOLVED)
Distinct from the resting-calm problem above: under HARD stress (jetting water at terrain, deep hydrostatic columns) the clamped weakly-compressible EOS let particles collapse into over-dense knots it turned into a perpetual bounce engine — the owner's "infinite energetic mass / giant particles that never shrink and bounce forever." Solved in layers, NOT by more damping (damping/viscosity only MASK it, confirmed across ~15 versions): **`LIQUID_DENS_CAP`** (v24.182, edit²) clamps the density the pressure pass sees so a knot's impulse is bounded; **the anti-clump min-separation pass** (`WGSL_DECLUMP` in `liquid-wgpu.js`, v24.185, gm `water.DECLUMP`) pushes over-dense neighbours apart each substep so the over-packing can't persist; **v24.186** clamps that push against the terrain mask so it never bleeds water into solid tiles. `RAW`/`SUBSTEP_DT`/`GRAVITY` (v24.169) set the calm baseline; `MAX_VEL`/`BURST_DAMP` (v24.173) are bounded safeguards. Full lever table + rationale in `TUNING.md` §2.10. **The pondtest harness CANNOT reproduce the dug-cavern runaway** (every pond is a dense body below the 6.4 declump trigger), so verify safety (self-tests green, resting pond unchanged) in-harness and efficacy via the owner's dev `X` overlay (`dn MAX` + `v: mean` lines). RULE learned: any positional pass that moves particles must be terrain-aware, and "this shouldn't happen at all" means a missing invariant — remove the source, don't damp the symptom.

## Workflow: Opus plans, Sonnet executes

When tackling anything non-trivial, split the work by model capability rather than letting one model do everything. The pattern is: use **Claude Opus** for the thinking work (planning, architecture decisions, designing the approach, identifying the right place to make a change), then hand off to **Claude Sonnet** to carry out the plan (writing the actual code, making the edits, running tasks, verifying the result).

In practice this means: if you're about to make a structural decision — where to add a new system, how to refactor a section, what the fix strategy for a tricky bug should be — stop and use Opus first. Get a clear plan. Then switch to Sonnet and execute it step by step.

This isn't bureaucracy; it's economics. Opus is slower and more expensive but thinks more carefully about the whole problem. Sonnet is faster and cheaper and very good at executing a well-defined task. Mixing them deliberately gets better results than either alone, and matters especially for a file like `sluice.js` where a bad architectural call is expensive to undo.

## Parallelism preference

When a task has independent, separable parts, strongly prefer launching them as **concurrent subagents in a single message** rather than working through them sequentially. Decompose multi-part requests into the maximum number of independent subtasks and dispatch them in parallel — the more genuinely independent the pieces, the more agents run at once.

Caveat for this repo: parallelize **read-only / research** work freely (exploring the game, gathering context, reviewing, hunting where something is defined). For **edits**, the split helps: concurrent subagents editing **different** `js/sluice/` fragments is now safe, since each writes its own file. But never have two agents edit the **same** fragment (overlapping edits clobber), and never let an agent edit the generated `js/sluice.js` bundle at all; rebuild it once with `build-sluice.sh` after the fragment edits land. If a change spans many fragments at once, keep it sequential.

## Versioning & save rules

**After any change to `sluice` files:**

1. **Edit a fragment, not the bundle.** Game code lives in `js/sluice/NNN-*.js` (see "Split layout"); `js/sluice.js` is generated. Never hand-edit the bundle; `build-sluice.sh` overwrites it.
2. **Bump the version number.** Every edit (bug fix, tweak, new feature) increments the version. `GAME_VERSION` lives in `js/sluice/000-head.js` (grep `GAME_VERSION`); update it before calling the work done.
3. **Rebuild the bundle.** Run `./build-sluice.sh` to regenerate `js/sluice.js` from the fragments.
4. **Commit to local `main` AND push, every change (owner rule).** You are editing the owner's one checkout directly, so the commit must land on their local `main` and on the remote, every single time. Stage path-scoped (the fragment(s) + the rebuilt `js/sluice.js`, never `git add -A`), commit on `main`, then `git pull --rebase origin main` (origin churns, other accounts share this remote) and `git push origin main`. After pushing, verify `git rev-parse main` equals `git rev-parse origin/main` so the owner's checkout is not left diverged. This applies to *every* change, including minor increments; a change isn't done until local `main` and the remote both have it. **Push only coherent, parseable builds**, never leave the `main` tip mid-refactor. The repo is **not** auto-deployed anywhere.
